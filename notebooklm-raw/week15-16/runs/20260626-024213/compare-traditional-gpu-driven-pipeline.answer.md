以下是基于 **Week16 笔记原稿** 与 **GPU驱动渲染管线(GPU-Driven Rendering Pipelines)** 论文素材整理的传统管线与现代管线对比指南。

### 1. 核心术语定义与背景
*   **绘制调用 Draw call**：CPU 向 GPU 发送指令，要求其渲染一组特定几何图元的过程 [1, 2]。
*   **CPU 剔除 CPU Culling / LOD(Level of Detail，细节层次)**：在应用阶段由 CPU 根据视锥体范围或物体距离，决定哪些物体需要渲染以及使用何种精度的模型 [3, 4]。
*   **MDI(Multi-Draw Indirect，间接多绘制)**：一种渲染指令，允许 GPU 独立根据其内部生成的参数渲染多个几何体，而无需 CPU 逐个干预 [5, 6]。
*   **网格簇 Meshlet**：将传统的复杂网格(Mesh)拆分成的固定拓扑、包含少量顶点（如 64 个）的小型簇 [4, 5]。
*   **网格着色器 Mesh Shader**：运行在 GPU 上的可编程着色器阶段，直接处理网格簇并自主执行剔除与 LOD 决策 [4]。
*   **延迟着色 Deferred Shading**：将几何处理与光照计算分离的技术，通常依赖 **G-Buffer(Geometry Buffer，几何缓冲区)** 存储像素信息 [4, 7]。
*   **可见性缓冲 Visibility Buffer**：现代渲染技术，仅渲染包含物体 ID 和深度信息的轻量级缓冲，用于后续查询材质并着色 [4]。

### 2. 传统 CPU-driven 与 GPU-driven 渲染管线对比

| 对比维度 | 传统 CPU-driven 渲染管线 | GPU-driven 渲染管线 |
| :--- | :--- | :--- |
| **调度核心** | **CPU 主导**。负责场景管理、物理模拟及每帧的渲染决策 [2]。 | **GPU 主导**。GPU 控制实际渲染的对象及“绘制场景”指令 [8, 9]。 |
| **绘制指令(Draw Call)** | 频繁的绘制调用。每个物体通常对应一个 Draw call，物体数量多时会导致 **CPU-GPU 通信瓶颈** [4]。 | **Multi-Draw**。单次 GPU 指令即可处理大量网格，由 GPU 动态生成绘制参数 [5, 10]。 |
| **剔除与 LOD** | **CPU 剔除/LOD 选择**。在应用阶段进行视锥体或遮挡剔除，受限于 CPU 的串行或有限并行能力 [3, 4]。 | **GPU 剔除**。基于 **网格簇(Meshlet)** 在 GPU 上执行细粒度的视锥体、遮挡及背面剔除 [4, 5]。 |
| **几何处理粒度** | **物体(Object)级**。以整个网格为基本单位，难以实现极高精度的局部裁剪 [4]。 | **网格簇(Meshlet)级**。通过 **网格着色器(Mesh Shader)** 实现极细粒度的几何控制与动态精简 [4, 5]。 |
| **着色架构倾向** | 经典的延迟着色(Deferred Shading)。依赖大量 **G-Buffer** 存储材质数据，显存带宽开销大 [4, 11]。 | **可见性缓冲(Visibility Buffer)**。放弃庞大的 G-Buffer，仅记录可见性 ID，后续按需对着色进行重构 [4]。 |
| **资源绑定** | 显式管理内存与资源绑定。开发者需频繁为每个 Draw call 切换纹理和状态 [12, 13]。 | **Bindless(无绑定)** 与虚拟纹理。所有纹理数据在单次绑定中均可访问，减少状态切换开销 [10, 14]。 |

### 3. 关键技术关系解析

#### 绘制调用(Draw call) 与 MultiDrawIndirect 的演进
在传统管线中，CPU 必须为每个可见物体准备状态并发出 Draw call，这在处理包含数万个小构件的复杂场景时会产生巨大的 API 开销 [4]。**MultiDrawIndirect** 允许 CPU 发出一条“渲染整个场景”的宏观指令，由 GPU 在着色器中根据剔除结果动态填充绘制参数缓冲区，从而彻底解耦 CPU 对渲染数量的限制 [5, 6, 8]。

#### CPU 剔除/LOD 与 Mesh Shader/Meshlet 的重构
传统方式在 CPU 端做粗略的物体级 LOD 切换，导致即使物体只有一小部分可见，GPU 仍需处理整个网格 [4]。**Mesh Shader** 架构将几何体离线预处理为 **Meshlets**，GPU 可以在渲染时实时判断每个簇的可见性。如果某个簇在视锥体外或被遮挡，GPU 会直接丢弃它，这种硬件级、细粒度的处理使 LOD 选择更平滑且高效 [4, 5]。

#### 延迟着色 与 Visibility Buffer 的关系
**延迟着色(Deferred Shading)** 虽然解决了光照复杂度问题，但由于需要 G-Buffer 存储法线、反照率等，导致高分辨率下的显存带宽压力巨大 [4, 11]。**Visibility Buffer** 是对该流程的现代优化：它不再存储材质属性，而只记录“哪个三角形的哪个部分在当前像素可见”(物体 ID + 原始三角形 ID) [4]。在着色阶段，GPU 再根据这些 ID 到虚拟纹理缓存中查询数据 [4, 10]。这种方式极大地优化了遮挡剔除效率，并降低了对 G-Buffer 带宽的依赖 [4, 7]。
