从固定功能管线（Fixed-function Pipeline）向可编程管线（Programmable Pipeline）的演进是计算机图形学硬件架构的重大变革，这一过程不仅赋予了开发者极大的灵活性，也显著提升了渲染的真实感 [1]。

### 1. 固定管线与可编程管线的差异

*   **固定功能管线（80年代 - 2000年初）**：以 1980 年代 Jim Clark 提出的“几何处理器”（Geometry Engine）为标志，硬件专门用于加速矩阵变换、裁剪等固定操作 [1]。在此阶段，GPU 像一个“黑盒”，开发者只能通过配置开关（如开启/关闭灯光）来影响渲染，无法自定义计算逻辑 [1, 2]。
*   **可编程管线（2001年至今）**：从 DirectX 9 引入顶点着色器（Vertex Shader）和像素着色器（Pixel Shader）开始，管线变得可编程 [1]。开发者可以通过编写着色器代码（Shader）来控制渲染逻辑。
*   **现代 API（Vulkan/DX12）**：Vulkan 和 DirectX 12 将这种灵活性推向极致，不仅管线阶段高度可编程，开发者还必须显式负责**系统级资源管理**（如内存分配、描述符集绑定、显存同步），这与旧式 API（如 OpenGL）的自动驱动管理有本质区别 [1, 2]。

### 2. 着色器（Shader）在管线中的位置

根据现代图形架构（如 DX11/12 和 Vulkan），着色器在管线中的工程位置如下 [3, 4]：

1.  **顶点着色器 (Vertex Shader)**：位于管线前端。处理输入的顶点数据，负责坐标变换 [3, 4]。
2.  **曲面细分着色器 (Tessellation Shader)**：位于顶点处理之后。它包含 Hull Shader、固定功能的细分器和 Domain Shader，用于动态增加几何细节（如地形渲染）[3-5]。
3.  **几何着色器 (Geometry Shader)**：处理完整的图元（如三角形）。它可以销毁或生成新的顶点，常用于粒子系统或阴影生成 [3, 4]。
4.  **像素/片元着色器 (Pixel/Fragment Shader)**：位于光栅化之后。它接收插值后的片元数据，计算最终的像素颜色、进行纹理采样和光照计算 [3, 4]。

### 3. 核心概念与 GPU 阶段的映射关系

现代管线中，开发者通过 Shader 手动实现数学逻辑，而硬件保留了部分高效的固定功能阶段 [3, 6]：

| 渲染概念 | 管线阶段 / 控制者 | 关系说明 |
| :--- | :--- | :--- |
| **MVP 变换** | **Vertex Shader** / 开发者 | 开发者需在顶点着色器中手动编写矩阵乘法（Model-View-Projection），将 3D 顶点转换到裁剪空间 [2, 4, 6]。 |
| **裁剪 (Clipping)** | **固定功能阶段** / 硬件 | 发生在投影变换后、光栅化前。硬件根据视锥体自动剔除视野外的几何图元，这是由硬件自动执行的高效阶段 [3, 6, 7]。 |
| **光栅化 (Rasterization)** | **固定功能阶段** / 硬件 | 硬件将 2D 图元（如三角形）离散化为像素位置的片元流，这是 GPU 内部不可编程的硬连线逻辑 [3, 8]。 |
| **隐藏面消除 (Hidden Surface / Z-Buffer)** | **固定功能阶段** / 硬件 | 在管线末端（Output Merger 阶段），硬件自动执行深度测试（Z-Buffer 算法），通过比较深度值决定哪些像素可见，解决遮挡问题 [8, 9]。 |

### 4. 现代 API 中的 Shader 工程实践（以 Vulkan 为例）

在现代工程中，Shader 的位置不仅是逻辑单元，更是资源管理的中心：
*   **手动矩阵计算**：由于管线不再提供内置变换矩阵，开发者必须在 CPU 端构建旋转、缩放等矩阵，并通过 **Push Constants** 或 **Uniform Buffers** 传递给 Shader [2, 10]。
*   **资源绑定**：开发者需要通过 **Descriptor Sets**（描述符集）将纹理、缓冲区等资源显式绑定到 Shader 阶段 [1, 3]。
*   **着色器编译**：现代 API 要求 Shader 预编译为字节码（如 SPIR-V），并在创建图形管线对象（Pipeline Object）时显式指定各个 Shader 模块 [2]。
